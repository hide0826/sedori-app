## 3-6-9システム概要

このドキュメントは、HIRIOアプリ内で何度も登場する **「3-6-9システム」** の考え方と実装ポイントをまとめたものです。  
Cursor から参照しやすいように、ルールと関連コードの場所を一箇所に集約しています。

---

## コンセプト

- **目的**: 仕入れごとのリスク・確度を「3 / 6 / 9」の3段階で表現し、  
  - SKU  
  - 仕入一覧（開発タブ）  
  - 統計（仕入健全度・実効見込み利益）  
 で一貫して扱うための仕組み。
- **入力場所**: 仕入CSVの **`コメント` 列**（または `comment`）の先頭に数字やコードを入力。
- **主な使い道**:
  - SKU文字列への埋め込み（テンプレート `{rule369}`）
  - 仕入管理（開発）タブの `3-6-9` 列自動入力
  - 「仕入健全度 (件数3-6-9 / 金額3-6-9)」と実効見込み利益の計算

---

## コメント → 3P/3N/6P/6N/9P/9N 変換ルール

実装は `python/services/sku_template.py` の `_get_rule369_code()` にあります。  
`コメント` の**先頭トークン**だけを見て、以下のように判定します（大文字小文字は無視）。

- `3` → `3P`
- `4`, `3n` → `3N`
- `6` → `6P`
- `7`, `6n` → `6N`
- `9` → `9P`
- `10`, `9n` → `9N`
- 空欄や上記以外（`11`, `12` など連番を含む場合も） → `6P`

この結果が **3-6-9コード**（`3P/3N/6P/6N/9P/9N`）です。

---

## 実運用イメージ（3 / 6 / 9 の意味）

数値の大きさはおおよそ以下のイメージで運用する想定です（厳密な意味付けは運用側で調整可能）。

- **3グループ（3P / 3N）**:  
  確度が高い・鉄板寄りの仕入れ。長期在庫リスクは低い想定。
- **6グループ（6P / 6N）**:  
  標準的な仕入れ。通常リスク〜やや注意レベル。
- **9グループ（9P / 9N）**:  
  攻めている仕入れ・長期在庫リスクが高め。  
  FBA長期在庫や在庫過多になりやすいゾーンを意識するためのラベル。

コメントに書くのは **先頭だけ** でよく、後ろにメモを続けても問題ありません（例: `3n-テスト`, `9 長期在庫候補` など）。

## 関連実装①: SKUテンプレート `{rule369}`

- ファイル: `python/services/sku_template.py`
- クラス: `SKUTemplateRenderer`
- ポイント:
  - SKUテンプレート文字列内で `{rule369}` を使うと、上記ルールで算出した `3P/3N/6P/6N/9P/9N` が挿入されます。
  - デフォルトテンプレート（`config/inventory_settings.json`）は  
    `{date:YYYYMMDD}-{supplier}-{purchasePrice}-{rule369}-{seq:3}`  
    のように、3-6-9コードをSKUの一部として利用します。

---

## 関連実装②: 仕入管理（開発）タブの「3-6-9」列

- ファイル: `python/desktop/ui/inventory_widget.py`
- 挙動（`dev_mode=True` の開発タブのみ）:
  - CSV取込時に、仕入データ一覧に `3-6-9` 列を追加。
  - `コメント` / `comment` 列を読み取り、  
    `sku_template.py` と同じルールで 3-6-9コードを推定し、自動で `3-6-9` 列に反映。
  - すでに `3-6-9` 列に値が入っている場合は **上書きしない**（ユーザー編集優先）。

## 関連実装③: 仕入健全度(3-6-9) と実効見込み利益

### 件数ベースの仕入健全度

- ファイル: `python/desktop/ui/inventory_widget.py`
- メソッド: `_compute_health_score_3_6_9`
- 内容:
  - `コメント` 列の先頭数字から、各行を 3 / 6 / 9 のいずれかに分類。
  - `3/6/9` の件数比率（%）を計算し、  
    - 9グループ比率 \(R9\) が
      - 〜15% → **安全圏**（緑）
      - 〜25% → **注意圏**（黄）
      - それ以上 → **危険域**（赤）
  - 結果は統計ラベル `仕入健全度(件数3-6-9)` として表示。

### 利益ベースの仕入健全度と実効見込み利益

- ファイル: `python/desktop/ui/inventory_widget.py`
- メソッド: `_compute_health_score_3_6_9_profit`
- 内容:
  - `コメント` の 3/6/9 分類と `見込み利益` を用いて、  
    各グループごとの理論見込み利益と「実効見込み利益」を計算。
  - 係数例:
    - 3グループ: 0.9
    - 6グループ: 0.7
    - 9グループ:
      - FBA長期在庫系 → 0.6
      - 自己発送など → 0.9
  - これにより
    - 「仕入健全度(金額3-6-9)」
    - 「実効見込み利益 / 想定見込み利益 / 実現率(%)」
    を統計エリアに表示します。

## 将来拡張: 在庫保管料を考慮した3-6-9モデル

3-6-9システムの最終ゴールの1つは、**ルート毎の「実効見込み利益」を、時間とともに実際の粗利に近づけていくこと**です。  
そのためには、将来的に次の2レイヤーで在庫保管料を扱う構想があります。

### レイヤー1: Keepaベースの概算保管料（理論モデル）

- **目的**: 実際のFBA請求レポートを使わずに、SKUごとの「期待保管コスト」をざっくり見積もる。
- 情報ソース:
  - `KeepaService`（`python/desktop/services/keepa_service.py`）経由で取得する
    - 商品サイズ（縦×横×高さ）
    - 重量
  - Amazon FBAの公開料金表（標準サイズ/大型サイズごとの保管料レート）をアプリ側にテーブルとして保持。
- モデル案:
  1. Keepaのサイズ情報から体積 \(V\) を算出し、FBAのサイズ区分を推定。
  2. サイズ区分ごとの「月次保管料単価」と掛け合わせて **月次保管料の理論値** を出す。
  3. 3-6-9ごとに想定滞留期間（例: 3グループ=3ヶ月, 6グループ=6ヶ月, 9グループ=9〜12ヶ月）を設定し、  
     \(\text{期待保管料} ≒ \text{月次保管料} × \text{想定滞留月数}\) として見込み利益から差し引く。
- 役割:
  - 「大きくて重いSKUを9に入れると、どれくらい保管コストが効いてくるか」を3-6-9に反映。
  - 実FBAレポートを使う前の **第1段階の保管料モデル** として利用。

### レイヤー2: 実際のFBA保管料レポートとの連携（精度向上用）

- **目的**: 実際にAmazonから請求された保管料を、ルート/3-6-9ごとの評価に反映させる。
- 情報ソース（将来案）:
  - Seller Centralの
    - FBA月次保管料レポート（`FBA Inventory Storage Fee`）
    - 長期在庫手数料レポート（`Long-Term Storage Fee`）
  - これらは多くの場合「SKU単位というより月次の集計行」で来るため、そのままではルートに結びつかない。
- 配分イメージ:
  1. 月次の保管料合計を取得。
  2. 在庫日数・在庫金額・体積などを重みとして、SKUごとに保管料を按分。
  3. SKUとルート・3-6-9分類は既存の仕入DB/ルートDBから解決し、**ルート×3/6/9ごとの実保管コスト**を集計。
  4. 「理論保管料（Keepaモデル）」と「実保管料（FBAレポート）」の差を見て、3-6-9ごとの係数や想定滞留月数をチューニング。

### ルート別学習のイメージ

- 各ルートごとに、以下を継続的に記録・比較する構想:
  - 想定見込み利益（仕入時の理論値）
  - 実効見込み利益（3-6-9係数＋期待保管料を反映した予測値）
  - 実粗利（決済レポート＋実保管料を含めた最終利益）
- これらの差分を元に、
  - ルート別
  - 3/6/9グループ別
に **係数やルール（滞留期間想定など）を自動調整していく** ことで、  
「使えば使うほど自分の実績にフィットしていく3-6-9モデル」を目指す。

## まとめ

- **入力**: 仕入CSVの `コメント` 列の先頭に 3 / 6 / 9（および n/p など）を書く。
- **変換**: `_get_rule369_code()` が `3P/3N/6P/6N/9P/9N` を返す。
- **利用先**:
  - SKUテンプレート `{rule369}`（SKU内に3-6-9コードを埋め込み）
  - 仕入管理（開発）タブの `3-6-9` 列
  - 仕入健全度(件数3-6-9 / 金額3-6-9) と実効見込み利益の計算
